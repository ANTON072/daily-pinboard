## アーキテクチャ概要

### 構成方針

- サーバーレス単一 Worker としてすべての処理を実装する
- 永続ストレージは Cloudflare D1 のみ使用する
- 外部サービスへの依存は OpenAI / Resend / Pinboard の3つに限定する
- フロントエンドなし（バッチ処理のみ）

### ランタイム制約

Cloudflare Workers Free プランの制約を前提とした設計とする。

| 制約項目         | 上限                                        | 対応方針                                           |
| ---------------- | ------------------------------------------- | -------------------------------------------------- |
| サブリクエスト数 | 50件/呼び出し                               | 2段階フィルタで記事フェッチを30件以内に抑える      |
| CPU 時間         | 30ms（HTTP リクエスト） / Cron は実質無制限 | Cron Trigger での起動のため CPU 時間制限は問題なし |
| メモリ           | 128MB                                       | 記事本文は先頭3,000文字に切り詰めて上限を回避      |
| リクエスト数     | 100,000件/日                                | Cron 起動のみのため上限に抵触しない                |

## データベース設計

### テーブル：`sent_articles`

送信済み記事URLを管理し、重複配信を防ぐ。

```sql
CREATE TABLE sent_articles (
  url     TEXT PRIMARY KEY,
  sent_at TEXT NOT NULL  -- ISO 8601 形式（例: 2026-05-07T00:00:00.000Z）
);
```

| カラム    | 型   | 説明                 |
| --------- | ---- | -------------------- |
| `url`     | TEXT | 記事URL（主キー）    |
| `sent_at` | TEXT | 送信日時（ISO 8601） |

**保持期間：** 90日。毎実行時に以下のクエリで期限切れレコードを削除する。

```sql
DELETE FROM sent_articles WHERE sent_at < datetime('now', '-90 days');
```

## 外部API仕様

### Pinboard Popular Feed

| 項目           | 内容                                                      |
| -------------- | --------------------------------------------------------- |
| エンドポイント | `https://feeds.pinboard.in/json/popular/`                 |
| メソッド       | GET                                                       |
| 認証           | なし                                                      |
| レスポンス     | JSON 配列（最大50件）                                     |
| 取得フィールド | `u`（URL）、`d`（title）、`n`（description）、`t`（tags） |

### OpenAI API

| 項目         | 内容                                                           |
| ------------ | -------------------------------------------------------------- |
| モデル       | `gpt-4o-mini`                                                  |
| 用途         | 第1段階スコアリング、第2段階スコアリング、日本語要約生成       |
| 呼び出し方式 | スコアリングはバッチ（一括）、要約生成は記事ごとに個別呼び出し |
| 認証         | Bearer トークン（`OPENAI_API_KEY` シークレット）               |

### Resend API

| 項目           | 内容                                             |
| -------------- | ------------------------------------------------ |
| エンドポイント | `https://api.resend.com/emails`                  |
| メソッド       | POST                                             |
| 送信元         | `onboarding@resend.dev`                          |
| 認証           | Bearer トークン（`RESEND_API_KEY` シークレット） |
| 形式           | プレーンテキスト                                 |

## 環境変数・シークレット

すべての機密情報は Cloudflare Workers のシークレットとして管理する。ソースコードへのハードコードは禁止。

| 変数名           | 種別         | 説明                 |
| ---------------- | ------------ | -------------------- |
| `OPENAI_API_KEY` | シークレット | OpenAI API キー      |
| `RESEND_API_KEY` | シークレット | Resend API キー      |
| `TO_EMAIL`       | シークレット | メール配信先アドレス |

## デプロイ構成

### Cron スケジュール

```toml
# wrangler.toml
[triggers]
crons = ["0 0 * * *"]  # UTC 00:00 = JST 09:00
```

```
ローカル開発
  └─ wrangler dev（ローカル動作確認）
       └─ git push → GitHub Actions
            └─ wrangler deploy（本番デプロイ）
```

- `main` ブランチへのプッシュ時に自動デプロイ
- `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` を GitHub Secrets で管理

## 技術的制約と要件

### セキュリティ

- APIキー類はすべて Cloudflare Workers のシークレットで管理する
- ソースコードへのキーのハードコードは禁止
- 外部フィードから取得したデータをメール本文に含める際、制御文字のサニタイズを行う

### 可用性

- Cloudflare Workers の SLA に依存（障害時は翌日の Cron 実行に委ねる）
- 記事フェッチ失敗時は該当記事をスキップし、処理を継続する
- Pinboard フィード取得失敗時のみ1回リトライを行う

### ログ・モニタリング

- すべてのエラーは `console.error` で出力し `wrangler tail` で確認する
- 将来的に永続ログが必要になった場合は D1 への記録を追加検討する

### テスト方針

- `wrangler dev` を使ったローカル統合テストを優先する
- 単体テストは複雑なロジック（スコアリング選出、フォールバック補充）に限定して導入する
